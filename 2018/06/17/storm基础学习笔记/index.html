storm基础学习笔记

发表于 2018-06-17 | 分类于 数据科学 | |
字数统计: 1,634 字 | 阅读时长 ≈ 6 分钟

快速介绍

  • 定义：
    storm 是一个开源的分布式实时计算系统， 可以简单、可靠的处理大量的数据流。
  • 特点：
    编程模型简单： 简单优美的原语
    可扩展： 拓扑： 工作进程 线程 任务
    高可靠
    高容错
    支持多种编程语言
    支持本地模式
    高效 zeroMQ

基本概念及名词

一共有两种节点：由 Zookeeper 协调 （快速失败 无状态）

  • 控制节点（master node） nimbus
  • 工作节点（worker node） supervisor

strom 上运行的是 拓扑（topology），在执行时可以看到拓扑图
Topologies 由一下几部分构成：

  • spouts
    • 消息生产者读取外部数据源向拓扑里发送消息
    • 可靠 或 不可靠
    • 重要的方法： nextTuple ack fail
  • blots 消息处理逻辑 主要方法是execute 以一个tuple作为输入
    • bolts使用OutputCollector来发射tuple emit
    • bolts必须要为它处理的每一个tuple调用OutputCollector的ack方法
    • 一般处理流程：
      • bolts处理一个输入tuple,
      • 发射0个或者多个tuple,
      • 然后调用ack通知storm自己已经处理过这个tuple了( ack 可自动调用)
  • stream消息流：没有边界的 tuple序列
  • Stream groupings： 定义每个blot是接受什么样的流作为输入。
    • shuffle grouping 随机分组： 随机选择一个tuple
    • fields grouping 按字段分组 只接受 某些‘user_id’ 的tuple
    • all grouing 广播， 任何一个tuple都会复制并发送个下游所有的blot
    • global grouing 全局分组： 所有tuple进入下游blot的同一个task,通常只选择 一个taskid最小的

tasks：

  • 每一个spout和bolt会被当作很多task在整个集群里执行
  • setSpout 可以设置并行度

workers:

  • 一个topology可能会在一个或者多个worker（工作进程）里面执行，每个worker是一个物理JVM并且执行整个topology的一部分

执行方式
storm jar all-my-code.jar backtype.storm.MyTopology arg1 arg2

核心实现逻辑

高可靠性（Reliability ）： storm 可以确保spout 发出来的每个消息都会被完整的处理

  • tuple tree
  • storm 判断一个spout发送出的消息被完整处理的条件：
    1. tuple tree 不在生长
    2. 数中的所有消息均被标识为 “已处理”
  • 消息的生命周期：
    1. spout实例调用 nextTuple() 从spout请求一个消息（tuple）
    2. 收到请求后， spout 使用 open方法 提供的 spoutOutputCollector向他的输出流发送一个或者多个消息
    3. spout会给发送的每一个消息提供一个message_id,用来标识这个消息
    4. 这个消息会被发送到后续业务处理的blot中，并且 storm会跟踪由此消息产生出来的新消息
    5. 当检测到一个消息衍生出来的整个 tuple tree 被完整处理后， storm会调用spout中的ack方法，参数为此消息的messageid（衍生的tuple会有messageid吗）
    6. 如果消息被处理超时，则消息对应的spout的fail 方法会被调用， 参数为messageid

      注： 一个消息只会由发送它的那个spout任务来调用ack 或fail. 如果系统中某个spout由多个任务运行，消息也只会由创建他的spout任务来应答（ack or fail）.决不会由其他的spout任务来应答

注：
我们先简述一下kestrel消息队列：
当KestrelSpout从kestrel队列中读取一个消息，表示它“打开”了队列中某个消息。这意味着，此消息并未从队列中真正的删除，而是将此消息设置为“pending”状态，它等待来自客户端的应答，被应答以后，此消息才会被真正的从队列中删除。处于“pending”状态的消息不会被其他的客户端看到。另外，如果一个客户端意外的断开连接，则由此客户端“打开”的所有消息都会被重新加入到队列中。当消息被“打开”的时候，kestrel队列同时会为这个消息提供一个唯一的标识。
KestrelSpout就是使用这个唯一的标识作为这个tuple的messageID的。稍后当ack或fail被调用的时候，KestrelSpout会把ack或者fail连同messageID一起发送给kestrel队列，kestrel会将消息从队列中真正删除或者将它重新放回队列中。

可靠相关的API:
为了使用storm提供的可靠处理特性，我们需要做两件事：

1. 无论何时在tuple tree 中创建一个新的节点， 我们需要明确的通知storm
2. 当处理完一个单独的消息时，我们需要告诉storm这颗 tuple tree 的变化

锚定（anchoring）: 为 tuple tree 中指定的节点增加一个新的节点

  • 消息锚定的方式：
    • 把输入消息作为emit方法的第一个参数_collector.emit(tuple, new Value(word))
  • 不锚定的方式：
    • _collector.emit(new Value(word))

锚定的规则：

  1. 一个输出消息可以被锚定在一个或者多个输入消息上
  2. 每个被处理的消息必须表明成功或失败
    很多bolt遵循特定的处理流程： 读取一个消息、发送它派生出来的子消息、在execute结尾处应答此消息

高效的实现 tuple tree:
storm 系统中有一组叫做“acker” 的特殊任务,他们负责跟踪DAG中的每个消息
每当发现一个DAG被完全处理，他就向创建这个根消息的spout任务发送一个信号。

消息的生命周期：

  1. 当一个消息被创建时（无论在soput还是在blot中），系统都为该消息分配一个64位的随机id,
    acker 用这些id来跟踪 由spout 消息派生出来的tuple tree
    2.每个消息都知道它所在的根消息的id:
    每当blot新生成一个消息，对应的 tuple tree 中的根消息的messageid 就拷贝到这个消息中

  2. 当这个消息被应答的时候，他就把关于tuple tree 变化的信息发送给 跟踪这棵树的 acker

  3. tuple 跟踪算法：
    storm 只需要使用固定内存（20字节）就可以跟踪一颗树
    acker任务保存了spout的消息id : 值的映射。

     第一个只是 spout的任务id
 第二个是 一个64 bit的数字， （ack val） 是树中所有消息id 的异或结果
当消息被创建和被应答的时候都会有相同的消息id发送过来做异或。

    每当acker发现一棵树的ack val值为0的时候，它就知道这棵树已经被完全处理了
